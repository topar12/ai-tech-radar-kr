API 연결 가이드

정적 프론트와 별도 백엔드 서버를 안정적으로 붙이기 위한 운영 계약입니다.

핵심 결정 현재 구현 권장 구조 연결 방법 응답 계약 필요한 키 수집 정책 백엔드 순서 체크리스트 참고 링크

AI Tech Radar KR · 2026-05-21

프론트는 정적으로 두고, 데이터는 별도 백엔드 서버가 책임진다.

현재 앱은 기본적으로 샘플 데이터를 쓰지만, URL 파라미터나 localStorage로 API 주소를 넘기면 GET /api/bootstrap 응답을 받아 화면 데이터를 교체합니다. 장기 운영 기준에서는 이 API가 실시간 계산을 하지 않고, 백엔드가 미리 만든 최신 스냅샷을 빠르게 내려주는 구조가 가장 안전합니다.

이번 결정 한눈에

백엔드는 별도 서버

GitHub 저장소를 운영 백엔드처럼 쓰지 않고, FastAPI 서버가 수집과 DB를 맡습니다.

프론트는 스냅샷만 읽음

브라우저는 GET /api/bootstrap으로 이미 만들어진 최신 레이더만 받습니다.

운영 DB는 Postgres

SQLite는 로컬 개발과 초기 실험용, 공개 장기 운영은 Postgres를 기준으로 둡니다.

사용자 알림은 보류

이메일/푸시보다 홈과 Watchlist에서 상태 변화를 잘 보여주는 것을 먼저 만듭니다.

현재 구현 상태

프론트 fallback

API 주소가 없거나 실패하면 기존 샘플 데이터로 렌더링합니다. 화면은 깨지지 않습니다.

별도 백엔드 서버

실수집, DB, 키 관리, 관리자 작업은 브라우저가 아니라 서버가 맡습니다.

Snapshot API

/api/bootstrap 하나로 이슈, 신호, 출처, Watchlist, 라벨 사전을 한번에 받습니다.

Mock API

api/mock-radar-server.js로 실제 연결 흐름을 바로 테스트할 수 있습니다.

정적 HTML 유지 FastAPI 권장 캐시된 스냅샷 사용자 주기 알림 보류

최신 권장 구조

GitHub를 백엔드처럼 쓰는 방식은 초기에 빠르지만, 키 보관, 수집 로그, 재시도, 관리자 실행, DB 백업을 장기적으로 다루기 어렵습니다. 따라서 공개 운영 기준은 정적 프론트와 별도 백엔드 서버를 분리합니다.

Static Frontend

GitHub Pages, Vercel, Netlify 어디든 배포 가능합니다. 브라우저에는 공개 API 주소만 노출합니다.

FastAPI Backend

수집기 실행, 스냅샷 생성, CORS, 관리자 인증, 헬스체크를 담당합니다.

Postgres Production

운영 DB는 Postgres를 기본으로 봅니다. 로컬 개발과 아주 초기 실험만 SQLite로 충분합니다.

Snapshot Builder

수집된 원천 데이터를 점수화하고 /api/bootstrap용 읽기 모델을 미리 만들어 둡니다.

Browser
  -> GET /api/bootstrap
FastAPI
  -> latest_radar_snapshot
  -> Postgres tables: sources, signals, issues, jobs
Collectors
  -> GitHub, Hugging Face, arXiv, Hacker News, RSS/Atom

사용자를 깨우는 주기적 알림은 지금 범위에서 제외합니다. 다만 데이터 신선도를 위해 관리자 수동 수집과 서버 내부 스케줄 수집은 필요합니다.

연결 방법

1. 로컬 Mock API 서버로 테스트

cd ai-tech-radar-kr
node api/mock-radar-server.js

2. 프론트 정적 서버 실행

cd ai-tech-radar-kr
python3 -m http.server 8765

3. API 주소를 붙여서 열기

http://127.0.0.1:8765/index.html?api=http://127.0.0.1:8787

대체 연결 방식

자주 쓸 API 주소는 브라우저 콘솔에서 한번 저장해둘 수 있습니다.

localStorage.setItem("radarApiBaseUrl", "http://127.0.0.1:8787");

배포 HTML에서 고정하려면 app.js보다 먼저 아래 전역 값을 선언하면 됩니다.

<script>
  window.RADAR_API_BASE_URL = "https://api.your-radar.com";
</script>

실제 운영에서는 프론트 도메인과 API 도메인을 분리해도 됩니다. 예: https://radar.example.com + https://api.radar.example.com.

필수 API 계약

공개 화면은 /api/bootstrap 하나만 정확히 맞추면 살아납니다. 관리자 기능은 별도 보호 엔드포인트로 분리하고 브라우저에서 직접 호출하지 않습니다. 

GET /api/bootstrap
Accept: application/json

엔드포인트 최소 세트

엔드포인트 공개 여부 역할
GET /health공개배포 상태와 DB 연결 상태를 빠르게 확인합니다.
GET /api/bootstrap공개최신 레이더 스냅샷을 반환합니다. 요청 시점에 무거운 수집/요약을 실행하지 않습니다.
GET /api/issues/:id선택 공개상세 화면이 커지면 이슈 상세만 따로 받을 수 있습니다.
POST /api/admin/collect보호관리자가 수집 작업을 수동 실행합니다.
POST /api/admin/rebuild-snapshot보호현재 DB 기준으로 /api/bootstrap 스냅샷을 다시 만듭니다.
GET /api/admin/jobs보호최근 수집 작업, 실패 이유, 실행 시간을 확인합니다.

Issue 필드

필드 필수 설명
id예이슈 고유 ID. 저장/숨김 상태가 이 값을 씁니다.
title예카드 제목. 사건명이 아니라 변화 단위로 작성합니다.
conclusion예상세 최상단에 보이는 결론 문장입니다.
categories예models, agents, infra 같은 배열입니다.
tags예검색과 질문형 탐색에 쓰는 키워드입니다.
importance, velocity, practicalValue, koreaRelevance예0-100 점수. 정렬 탭에서 씁니다.
summary예whatHappened, whyMatters, whoAffected, nextAction가 필요합니다.
sourceIds, signalIds예출처와 신호 배열을 연결하는 ID입니다.
timeline, validation예상세 탭의 흐름/행동 영역에 표시됩니다.

CORS 헤더

프론트와 API 포트가 다르면 CORS가 필요합니다. 개발 중에는 전체 허용으로 시작해도 되지만, 운영에서는 프론트 도메인만 허용합니다. 

Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, OPTIONS
Access-Control-Allow-Headers: Content-Type, Accept

내가 연결해야 할 수 있는 API 키

용도 키 필수 여부 메모
GitHub 레포/토픽 활동 GITHUB_TOKEN 권장 무토큰도 가능하지만 rate limit이 낮습니다. 조건부 요청과 backoff를 같이 씁니다.
Hugging Face 모델 동향 HF_TOKEN 권장 공개 모델 조회는 일부 무토큰 가능. 안정성을 위해 토큰 권장.
LLM 요약/클러스터링 OPENAI_API_KEY 등 선택 처음에는 규칙 기반 점수화로 시작하고, 요약 품질 개선 시 붙입니다.
관리자 실행 보호 ADMIN_TOKEN 필수 수집 실행, 스냅샷 재생성 같은 보호 엔드포인트에만 씁니다.
운영 DB 연결 DATABASE_URL 운영 필수 공개 운영은 Postgres 권장. 로컬은 SQLite 파일로 시작합니다.
스케줄러 보호 CRON_SECRET 선택 외부 cron이 수집 API를 때리는 구조라면 별도 비밀값을 둡니다.

주의: 유료 API 키는 클라이언트 HTML에 넣지 말고 반드시 서버 환경변수로만 보관해야 합니다.

수집 정책

빠른 실시간성보다 안정적인 최신성이 더 중요합니다. 각 소스의 공개 정책과 rate limit을 지키고, 실패한 수집은 사용자 화면이 아니라 관리자 작업 로그에서 먼저 드러나야 합니다.

소스 권장 주기 운영 원칙
GitHub1-6시간토큰, 조건부 요청, 403/429 backoff. Trending 스크래핑은 보조로만 둡니다.
Hugging Face30-60분토큰 권장. 모델 카드, 라이선스, 다운로드/좋아요 신호를 분리 저장합니다.
arXiv일 1-2회요청 간격을 넉넉히 두고, 논문은 당일 이슈보다 배경 신호로 취급합니다.
Hacker News30-60분공식 Firebase API 또는 Algolia API를 사용하고, 점수보다 토론 맥락을 봅니다.
RSS/Atom피드 TTL 기준ETag, Last-Modified, robots.txt, 피드별 TTL을 존중합니다.

백엔드 구현 순서

  1. FastAPI 골격: /health, /api/bootstrap, CORS, 설정 로더, 로깅을 먼저 만듭니다.
  2. DB 모델: sources, signals, issues, issue_sources, jobs, snapshots를 둡니다.
  3. 수집 어댑터: RSS/Atom, GitHub, Hugging Face, arXiv, Hacker News를 독립 모듈로 구현합니다.
  4. 정규화: 원문을 보존하고 공통 Source/Signal 형태로 변환합니다.
  5. 클러스터링과 점수화: 같은 사건을 Issue로 묶고 중요도, 확산, 실무성, 국내 관련성, 리스크를 계산합니다.
  6. 스냅샷 생성: 최신 이슈 묶음을 snapshots에 저장하고 /api/bootstrap은 그 결과만 반환합니다.
  7. 관리자 운영: 수동 수집, 재빌드, 실패 로그, 백업/복구 절차를 붙입니다.

DB 선택

단계 권장 DB 이유
로컬 개발SQLite설치가 가볍고 파일 하나로 재현이 쉽습니다. WAL 모드를 켜면 개발 중 읽기/쓰기 충돌도 줄일 수 있습니다.
초기 비공개 운영SQLite 또는 Postgres수집 주기가 낮고 단일 프로세스면 SQLite도 가능하지만, 백업/마이그레이션 기준은 미리 잡습니다.
공개 장기 운영Postgres동시성, 백업, 마이그레이션, 관리 도구, 확장성을 생각하면 Postgres가 기본값입니다.

연결 전 체크리스트

  • GET /health가 200을 반환한다.
  • GET /api/bootstrap가 캐시된 스냅샷 JSON을 반환한다.
  • issues, sources, signals, watchlists가 모두 배열이다.
  • 모든 issue.sourceIds와 issue.signalIds가 실제 ID를 가리킨다.
  • 운영 CORS가 프론트 도메인만 허용한다.
  • 브라우저 번들에 GITHUB_TOKEN, HF_TOKEN, OPENAI_API_KEY가 노출되지 않는다.
  • 관리자 엔드포인트는 ADMIN_TOKEN 없이 실행되지 않는다.
  • 수집 작업의 성공/실패/소요 시간이 jobs 또는 로그에서 확인된다.
  • GitHub/HF/arXiv/RSS 요청에 rate limit, backoff, 조건부 요청 정책이 있다.
  • Postgres 백업 또는 SQLite 파일 백업 절차가 문서화되어 있다.
  • 프론트 URL에 ?api=...를 붙였을 때 상단 상태가 API 연결로 바뀐다.
  • API가 죽었을 때 샘플 데이터로 fallback 되는지 확인한다.
