API 연결 가이드

정적 프로토타입을 실제 데이터 레이더로 바꾸기 위한 연결 계약입니다.

현재 구현 연결 방법 응답 계약 필요한 키 백엔드 순서 체크리스트

AI Tech Radar KR · 2026-05-21

프론트는 준비됨. 이제 API는 이 계약만 맞추면 된다.

현재 앱은 기본적으로 샘플 데이터를 쓰지만, URL 파라미터나 localStorage로 API 주소를 넘기면 GET /api/bootstrap 응답을 받아 화면 데이터를 교체합니다. 즉, 수집기와 DB를 나중에 붙여도 UI를 다시 뜯지 않도록 연결 지점을 먼저 잡아둔 상태입니다.

현재 구현 상태

프론트 fallback

API 주소가 없거나 실패하면 기존 샘플 데이터로 렌더링합니다. 화면은 깨지지 않습니다.

API bootstrap

/api/bootstrap 하나로 이슈, 신호, 출처, Watchlist, 라벨 사전을 한번에 받습니다.

Mock API

api/mock-radar-server.js로 실제 연결 흐름을 바로 테스트할 수 있습니다.

정적 HTML 유지 CORS 대응 샘플 fallback 실수집기는 다음 단계

연결 방법

1. Mock API 서버로 테스트

cd ai-tech-radar-kr
node api/mock-radar-server.js

2. 프론트 정적 서버 실행

cd ai-tech-radar-kr
python3 -m http.server 8765

3. API 주소를 붙여서 열기

http://127.0.0.1:8765/index.html?api=http://127.0.0.1:8787

대체 연결 방식

자주 쓸 API 주소는 브라우저 콘솔에서 한번 저장해둘 수 있습니다.

localStorage.setItem("radarApiBaseUrl", "http://127.0.0.1:8787");

배포 HTML에서 고정하려면 app.js보다 먼저 아래 전역 값을 선언하면 됩니다.

<script>
  window.RADAR_API_BASE_URL = "https://api.your-radar.com";
</script>

필수 API 계약

첫 단계에서는 엔드포인트를 많이 만들 필요 없습니다. 이 하나만 정확히 맞추면 현재 화면 전체가 살아납니다.

GET /api/bootstrap
Accept: application/json

Issue 필드

필드 필수 설명
id예이슈 고유 ID. 저장/숨김 상태가 이 값을 씁니다.
title예카드 제목. 사건명이 아니라 변화 단위로 작성합니다.
conclusion예상세 최상단에 보이는 결론 문장입니다.
categories예models, agents, infra 같은 배열입니다.
tags예검색과 질문형 탐색에 쓰는 키워드입니다.
importance, velocity, practicalValue, koreaRelevance예0-100 점수. 정렬 탭에서 씁니다.
summary예whatHappened, whyMatters, whoAffected, nextAction가 필요합니다.
sourceIds, signalIds예출처와 신호 배열을 연결하는 ID입니다.
timeline, validation예상세 탭의 흐름/행동 영역에 표시됩니다.

CORS 헤더

프론트와 API 포트가 다르면 CORS가 필요합니다. 개발 중에는 아래 정도면 충분합니다.

Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, OPTIONS
Access-Control-Allow-Headers: Content-Type, Accept

내가 연결해야 할 수 있는 API 키

용도 키 필수 여부 메모
GitHub 레포/토픽 활동 GITHUB_TOKEN 권장 무토큰도 가능하지만 rate limit이 낮습니다.
Hugging Face 모델 동향 HF_TOKEN 선택 공개 모델 조회는 일부 무토큰 가능. 안정성을 위해 토큰 권장.
LLM 요약/클러스터링 OPENAI_API_KEY 등 선택 처음에는 규칙 기반 점수화로 시작하고, 요약 품질 개선 시 붙입니다.
뉴스/RSS 대부분 불필요 선택 공식 블로그 RSS, arXiv, HN 같은 공개 소스부터 시작합니다.

주의: 유료 API 키는 클라이언트 HTML에 넣지 말고 반드시 서버 환경변수로만 보관해야 합니다.

백엔드 구현 순서

  1. 수집: 공식 블로그 RSS, GitHub, Hugging Face, arXiv, Hacker News부터 시작합니다.
  2. 정규화: 원문을 Source로 저장하고, 출처 유형과 신뢰도를 붙입니다.
  3. 신호 생성: 릴리즈, 스타 증가, 논문, 커뮤니티 급등 같은 이벤트를 Signal로 만듭니다.
  4. 클러스터링: 여러 Source/Signal을 하나의 Issue로 묶습니다.
  5. 점수화: 중요도, 확산, 실무성, 국내 관련성, 리스크를 계산합니다.
  6. 전달: /api/bootstrap으로 현재 레이더 스냅샷을 내려줍니다.

처음 DB 추천

초기에는 SQLite 하나로 충분합니다. 테이블은 sources, signals, issues, watchlists, issue_sources, issue_signals 정도면 됩니다.

연결 전 체크리스트

  • GET /health가 200을 반환한다.
  • GET /api/bootstrap가 JSON을 반환한다.
  • issues, sources, signals, watchlists가 모두 배열이다.
  • 모든 issue.sourceIds와 issue.signalIds가 실제 ID를 가리킨다.
  • CORS 헤더가 들어 있다.
  • 프론트 URL에 ?api=...를 붙였을 때 상단 상태가 API 연결로 바뀐다.
  • API가 죽었을 때 샘플 데이터로 fallback 되는지 확인한다.
